# 戦国領国録 Codex実装指示書

## 1. 文書の目的

このディレクトリには、『戦国領国録』をOpenAI Codexで段階的に実装するための指示書を配置する。

各Phaseは独立した実装契約として扱う。

Codexは一度に一つのPhase、または一つの明確なサブタスクのみを実装する。

---

## 2. 仕様の優先順位

実装時は以下の順序で仕様を確認する。

1. `docs/GDD/GDD.md`
2. `docs/design/data-design.md`
3. `docs/design/technical-design.md`
4. リポジトリ直下の `AGENTS.md`
5. 対象Phaseの実装指示書
6. 既存コードとテスト

上位文書と下位文書が矛盾する場合は、上位文書を優先する。

ただし、矛盾を独断で解消してはならない。

矛盾を発見した場合は実装を停止し、以下を報告する。

- 矛盾している文書
- 該当箇所
- 実装へ与える影響
- 最小限の修正案

ゲーム仕様の変更が必要な場合は、GDD更新後に実装する。

## 4. Phaseの依存関係

Phaseは原則として順番に実施する。

```text
Phase01
  ↓
Phase02
  ↓
Phase03
  ↓
Phase04
  ↓
Phase05
  ↓
Phase06
  ↓
Phase07
  ↓
Phase08
  ↓
Phase09
  ↓
Phase10
  ↓
Phase11
  ↓
Phase12
  ↓
Phase13
  ↓
Phase14
```

ただし、Phase内のサブタスクは依存関係が明確であれば分割可能とする。

後続Phaseを先行実装してはならない。

例外が必要な場合は、依存関係と理由を説明し、承認を得る。

---

## 5. 各Phaseの開始条件

対象Phaseを開始する前に、以下をすべて満たす必要がある。

- 前Phaseの完了条件を満たしている
- 前Phaseの必須テストが成功している
- 型チェックが成功している
- ビルドが成功している
- 未解決の重大エラーがない
- 仕様矛盾がない
- 対象Phaseに必要な設計文書が存在する
- 対象Phaseの実装対象と対象外が確認されている

開始条件を満たさない場合は、次Phaseへ進まない。

---

## 6. Codexの標準作業手順

Codexは各Phaseで以下の順序を守る。

### 6.1 調査

1. `AGENTS.md` を読む
2. 対象Phaseの指示書を読む
3. 関連するGDD箇所を読む
4. 関連するデータ設計書を読む
5. 関連する技術設計書を読む
6. 既存コードを確認する
7. 既存テストを確認する
8. 依存関係を確認する
9. 影響範囲を整理する

### 6.2 実装計画

実装前に以下を簡潔に整理する。

- 実装目的
- 変更対象
- 変更しない範囲
- 依存関係
- 想定リスク
- テスト方法
- 完了条件

### 6.3 実装

- 一度に一機能だけ実装する
- 既存設計に従う
- 型を先に定義する
- テストを同時に追加する
- 不要コードを削除する
- 一時的なデバッグコードを残さない
- GDDにない仕様を追加しない

### 6.4 検証

最低限、以下を実行する。

```bash
npm run lint
npm run typecheck
npm run test
npm run build
```

対象によって以下も実行する。

```bash
npm run test:e2e
```

### 6.5 完了報告

以下の順番で報告する。

1. 問題の整理
2. 実装方針
3. 実装内容
4. 影響範囲
5. テスト内容
6. 完了条件

## 7. 一度に実装する単位

Phase全体が大きい場合は、以下のように分割する。

例: Phase02 全国マップ

```text
02-01 マップScene基盤
02-02 城データ読込
02-03 城表示
02-04 街道表示
02-05 カメラ操作
02-06 城選択
02-07 React連携
02-08 テスト
```

Codexへは原則として一つのサブタスクのみ依頼する。

複数サブタスクを同時に実装する場合は、相互依存が強く、分割すると不整合が発生する場合に限る。

## 8. 実装対象外の扱い

縦切り試作では以下を実装しない。

- 全国300城
- 武将1,000人以上
- 海戦
- 水軍
- 婚姻
- 複雑な後継者争い
- 武将育成ツリー
- オンライン対戦
- ランキング
- ガチャ
- 広告
- クラウドセーブ
- 複数シナリオ販売

将来必要になりそうという理由だけで、先行実装してはならない。

拡張可能な構造は維持するが、未使用機能は作らない。

## 9. 技術上の必須ルール

- TypeScript strictを使用する
- `any`を使用しない
- Reactへゲームロジックを書かない
- Phaserへ権威的なゲーム状態を持たせない
- Zustandを巨大な単一Storeにしない
- マスターデータをコードへ直接書かない
- AIをブラックボックス化しない
- セーブデータにVersionを持たせる
- Webとモバイルの永続化をRepositoryで抽象化する
- ランダム処理は保存可能なRNG状態を使用する
- フレームレートとシミュレーション結果を分離する

## 11. Phase完了条件

各Phaseは以下をすべて満たした場合のみ完了とする。

- 指示書の実装対象がすべて動作する
- 実装対象外へ変更が及んでいない
- 型エラーがない
- lintエラーがない
- 必須テストが成功する
- ビルドが成功する
- 関連UIが確認されている
- ドキュメントが更新されている
- 不要コードが削除されている
- TODOが残っていない
- 仮データが本番コードへ残っていない
- 完了報告が事実と一致している

---

## 12. 不明仕様への対応

不明点を推測して実装してはならない。

不明仕様がある場合は以下を報告する。

```text
仕様不足:
- 対象機能:
- 該当GDD章:
- 不足している決定:
- 実装への影響:

選択肢:
A.
B.
C.

推奨案:
理由:
```

承認後にGDDまたは設計書を更新してから実装する。

テスト用仮値を使用する場合は `tests/fixtures/` 内に限定する。

## 13. 変更管理

ゲーム仕様を変更する場合:

```text
提案
  ↓
承認
  ↓
GDD更新
  ↓
データ設計書更新
  ↓
技術設計書更新
  ↓
Phase指示書更新
  ↓
実装
  ↓
テスト
```

技術実装のみを変更する場合:

```text
影響確認
  ↓
技術設計書更新
  ↓
実装
  ↓
テスト
```

## 14. Git運用

- 小さいコミットを作る
- 一つのコミットは一つの目的にする
- Conventional Commitsを使用する
- Phase番号を必要に応じて本文へ記載する
- 関係のない整形を混在させない
- 依存更新を機能実装へ混在させない
- mainへ直接大規模変更を入れない

例:

```text
feat(map): add castle rendering for phase 02
test(time): cover 4x simulation speed
fix(save): preserve last valid autosave
docs(codex): clarify phase 07 completion criteria
```

---

## 15. Codexへの禁止事項

- GDDを読まずに実装する
- 仕様を推測する
- 複数Phaseを同時に進める
- 未承認機能を追加する
- 必須テストを省略する
- テスト失敗を無視する
- 型エラーを型アサーションで隠す
- `any`を使用する
- TODOを残す
- 仮データを本番へ残す
- コードをコメントアウトして残す
- 既存機能を無断で変更する
- 完了していない作業を完了と報告する
- 不要なライブラリを追加する
- 巨大ファイルや神クラスを作る
