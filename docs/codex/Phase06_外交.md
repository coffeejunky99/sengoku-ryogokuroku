# Phase06 外交

## 1. 目的

勢力間関係と外交行動を実装し、関係変化と結果理由を表示可能にする。

---

## 2. 参照文書

実装前に以下を読む。

- `AGENTS.md`
- `docs/codex/README.md`
- `docs/GDD/GDD.md`
- `docs/design/data-design.md`
- `docs/design/technical-design.md`
- 直前Phaseの完了報告

GDDに存在しない仕様は実装しない。

---

## 3. 開始条件

- 直前Phaseが完了している
- `npm run lint` が成功する
- `npm run typecheck` が成功する
- `npm run test` が成功する
- `npm run build` が成功する
- 未解決の重大エラーがない
- 対象機能に必要な仕様がGDDに存在する

条件を満たさない場合は開始しない。

---

## 4. 実装対象

- 外交関係状態
- 友好
- 同盟
- 停戦
- 宣戦
- 臣従
- 共同出兵要求の基盤
- 外交結果
- 期限管理
- 理由コード
- 外交UI

---

## 5. 実装対象外

- 婚姻外交
- 人質
- 複雑な条約交渉
- オンライン外交
- 未確定の成功率や期間の独断決定

---

## 6. 実装順序

1. DiplomaticRelationState
2. pairKey正規化
3. 外交行動Command
4. 可否判定
5. 成功判定
6. 関係更新
7. 期限切れ処理
8. 理由コード
9. React UI
10. テスト

---

## 7. アーキテクチャ方針

- 外交関係は勢力ペアごとに一意管理
- Domainで可否・結果を決定
- UIは予測と理由を表示
- 期限処理はsimulation tickへ接続
- AI外交はPhase10で実装する

---

## 8. データ・状態管理

- alliedとat_warを同時に保持しない
- vassal時はoverlordClanId必須
- pairKeyは勢力順に依存しない
- 破棄・違約履歴を記録可能にする
- 成功率の正式値はGDD承認済みのみ

---

## 9. テスト内容

### Unit
- pairKey一意性
- 状態遷移
- 相互排他
- 期限切れ
- 臣従整合性
- 宣戦条件

### Integration
- UIから外交Command
- 関係更新
- 自動停止通知
- 理由表示

### E2E
- 外交画面
- 同盟提案
- 宣戦
- 停戦
- 臣従

---

## 10. 完了条件

- 外交関係が正しく表示される
- 行動可否と結果理由が表示される
- 状態矛盾がない
- 期限処理が動作する
- lint/typecheck/test/build/e2eが成功する

---

## 11. Codexへの禁止事項

- AI外交を先行実装する
- 成功率を乱数だけで決める
- 同盟と交戦を同時保持する
- 婚姻外交を追加する

---

## 12. 完了報告形式

1. 問題の整理
2. 実装方針
3. 実装内容
4. 影響範囲
5. テスト内容
6. 完了条件

満たしていない条件がある場合は未完了として報告する。
