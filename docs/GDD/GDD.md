# 1. ゲーム概要

## 1.1 コンセプト

『戦国領国録』は、日本の戦国時代を舞台に、大名として領国経営、外交、家臣団運営、軍事行動を行うリアルタイム戦略シミュレーションである。

プレイヤーは城を増やすだけではなく、領民、経済、家臣、同盟国、従属勢力を管理しながら天下統一を目指す。

ゲームの中心は以下の循環とする。

> *内政 → 収入増加 → 人材登用・軍備 → 外交 → 出兵 → 領土獲得 → 復興・統治*

## 1.2 ゲームの特徴

- 時間が継続する全国マップ

- 城ごとの領国経営

- マップ上を実際に移動する武将と軍団

- 同盟、臣従、従属大名を含む外交

- 一時停止可能なリアルタイム合戦

- 戦場地形と兵科相性

- 大名や武将の性格に基づくAI

- プレイヤーの行動によって分岐する歴史

- ガチャ、スタミナ、対人競争を使用しない

- 一人で繰り返し遊べる買い切り型設計

# 2. デザイン目標

## 2.1 プレイヤーに与える体験

プレイヤーに以下の感覚を与える。

- 小勢力を育てる達成感

- 武将を適材適所に配置する判断

- 外交によって不利な状況を覆す戦略性

- 軍団が全国マップを移動する臨場感

- 自分の判断によって歴史が変化する感覚

- 強大化した後も続く政治的・軍事的緊張

## 2.2 設計原則

1.  数値の種類ではなく、数値同士の関係で深さを作る。

2.  単純作業は自動化し、重要な判断をプレイヤーに残す。

3.  強い兵科を直接弱体化せず、地形や状況によって有効性を変える。

4.  AIの行動理由をプレイヤーが確認できるようにする。

5.  序盤、中盤、終盤で異なる課題が発生するようにする。

6.  大量の城や武将を追加する前に、小規模版の面白さを検証する。

# 5. ゲーム進行

## 5.1 開始手順

7.  タイトル画面を表示する。

8.  「新規開始」を選択する。

9.  シナリオを選択する。

10. プレイする大名家を選択する。

11. 大名家の特徴と難易度を確認する。

12. ゲームを開始する。

## 5.2 時間進行

ゲーム内時間はリアルタイムで進行する。

### 5.2.1 時間単位と初期状態

固定simulation stepは250ミリ秒とし、4 simulation tickでゲーム内時間が1日進む。

時間定数は`SIMULATION_STEP_MS = 250`、`SIMULATION_TICKS_PER_GAME_DAY = 4`とし、時間速度の型は`TimeScale = 0 | 1 | 2 | 4`とする。

縦切り試作の開始日は1561年9月1日（1561-09-01）とし、新規ゲーム開始時の初期`timeScale`は`0`（0倍速）とする。

ゲーム内日時は年、月、日を保持する。ゲーム内暦は一般的な西暦の月日・閏年規則を使用し、歴史的な旧暦の再現は縦切り試作の対象外とする。

### 5.2.2 一時停止と速度変更

使用可能な時間速度は以下とする。

| 速度 | 動作 |
|---|---|
| 0倍速 | ゲーム内時間を停止する |
| 1倍速 | 実時間1秒でゲーム内1日進む |
| 2倍速 | 実時間1秒でゲーム内2日進む |
| 4倍速 | 実時間1秒でゲーム内4日進む |

プレイヤーは時間速度を明示的に選択する。速度を変更しただけでは即時にゲーム内日付を進めない。

時間停止中はゲームシミュレーションのみ停止し、ゲーム内時間を進行させない。React UI、Phaser描画、全国マップのパン、ズーム、城選択などの操作は継続する。これにより、プレイヤーは時間進行に追われず意思決定できる。

### 5.2.3 重要通知による自動停止

以下の場合は自動的に一時停止できる。

- 宣戦布告

- 敵軍の領内侵入

- 合戦開始

- 城の包囲

- 大名または重要武将が死亡した

- 歴史イベント

- 資金が一定値未満

7条件は初期状態ですべて有効とし、設定画面で条件ごとに有効・無効を変更できる。有効な条件が発生した場合は時間速度を0倍速へ変更し、無効な条件では自動停止しない。

通知を閉じても以前の速度へ自動復帰しない。プレイヤーが時間速度を再選択するまで0倍速を維持する。

外交、戦闘、イベント、資金判定など、各条件を実際に発生させる処理はそれぞれの機能を扱う開発Phaseで実装する。

### 5.2.4 プレイヤーの意思決定による自動停止

以下の画面または操作を開始した場合は、時間速度を0倍速へ変更する。

- 内政コマンド選択

- 外交コマンド選択

- 軍事コマンド選択

- 武将への命令

- 選択肢を伴うイベント

- プレイヤーの意思決定を必要とする確認画面

画面や確認を閉じても以前の速度へ自動復帰しない。プレイヤーが時間速度を再選択するまで0倍速を維持する。

単なる情報表示、ツールチップ、軽微なトースト通知では原則として停止しない。どの通知を軽微とするかは各通知の正式仕様で定義する。

各画面は時間状態を直接変更せず、Application層の共通停止処理を経由して停止を要求する。

### 5.2.5 バックグラウンド移行

documentが非表示になった時点で時間速度を0倍速へ変更する。バックグラウンド滞在時間はゲーム内時間へ加算しない。

フォアグラウンド復帰後も0倍速を維持し、自動再開しない。バックグラウンド停止は設定で無効化できない安全動作とする。

Phase03の時間進行ではRNGを使用しない。RNGは乱数を最初に使用する開発Phaseで正式に導入する。

# 8. 城と内政

## 8.1 城の能力値

- 人口: 収入、徴兵、復興速度に影響する

- 農業: 基礎収入と人口維持に影響する

- 商業: 金銭収入に影響する

- 民忠: 一揆、逃亡、徴兵効率に影響する

- 治安: 盗賊、反乱、商業効率に影響する

- 城防御: 攻城戦での耐久力に影響する

- 訓練度: 兵士の戦闘能力に影響する

- 荒廃度: 収入、人口、民忠を低下させる

## 8.2 内政命令

### 農業開発

- 農業値を上昇させる

- 人口回復速度を上げる

- 担当武将の政治能力が高いほど効果が大きい

### 商業振興

- 商業値を上昇させる

- 月間収入を増やす

- 港や市場がある城では効果が増える

### 施し

- 民忠を回復させる

- 資金を消費する

- 戦後の城で重要となる

### 治安維持

- 治安を上昇させる

- 盗賊や反乱の発生率を下げる

### 城郭強化

- 城防御を上げる

- 資金と時間を多く消費する

### 軍事訓練

- 城内兵士の訓練度を上げる

- 担当武将の統率能力を参照する

### 徴兵

- 人口を消費して兵士を増やす

- 民忠が低い場合は効率が低下する

# 9. 施設

# 14. 攻城戦

## 14.1 攻城方法

### 包囲

- 時間をかけて城を降伏させる

- 攻撃側の損害が少ない

- 軍団資金を継続的に消費する

### 強攻

- 短時間で城を攻撃する

- 攻撃側と防御側の損害が大きい

- 城防御と人口が低下する

### 降伏勧告

- 城兵力

- 城主の忠誠

- 援軍の可能性

- 攻撃側との戦力差

以上を基に成功率を判定する。

## 14.2 落城後

落城後の城には以下が発生する。

- 荒廃

- 人口減少

- 民忠低下

- 治安悪化

- 施設破壊の可能性

占領後に復興が必要となる。

# 15. 合戦

## 15.1 基本形式

- リアルタイム進行

## 15.2 縦切り試作の兵科

- 足軽: 防御と戦線維持に優れる

- 騎馬: 平地での移動と突撃に優れる

- 鉄砲: 遠距離攻撃に優れるが接近戦に弱い

## 15.3 部隊命令

- 移動

- 攻撃

- 突撃

- 射撃

- 待機

- 後退

- 撤退

- 自動行動

## 15.4 戦闘要素

- 士気

- 疲労

- 正面攻撃

- 側面攻撃

- 背面攻撃

- 地形補正

- 兵科相性

- 武将能力

- 部隊人数

## 15.5 地形

- 平地: 騎馬の移動と突撃が有効

- 森林: 騎馬性能低下、鉄砲射程低下

- 山岳: 移動速度低下、防御側有利

- 河川: 渡河中の防御低下

- 狭路: 大部隊の展開を制限

- 城門: 防御側に高い防御補正

## 15.6 勝敗

以下の条件で合戦終了とする。

- 敵総大将が敗走する

- 敵軍の士気が0になる

- 敵軍が全軍撤退する

- 自軍が全軍撤退する

# 25. 非機能要件

## 25.1 パフォーマンス

- 対象端末で30FPS以上を維持する

- 通常時は60FPSを目標とする

- 画面切替は1秒以内を目標とする

- セーブは3秒以内を目標とする

- 10城、30武将、複数軍団が存在しても操作不能にならない

## 25.2 安定性

- セーブデータを破損させない

- 例外発生時にゲーム全体を停止させない

- 型エラーを残さない

- 未処理のPromiseエラーを残さない

- 主要ロジックには自動テストを作成する

## 25.3 モバイル対応

- iPhoneの安全領域へ対応する

- 縦画面を基本とする

- 合戦画面のみ横画面を検討する

- タッチ対象は十分な大きさを確保する

- 端末回転時に状態を破損させない

# 26. バランス方針

## 26.1 経済

- 武将数と軍団数が増えるほど維持費を増加させる

- 戦争によって領土が荒廃する

- 占領直後の城を即座に収入源にしない

- 小勢力にも外交による生存手段を与える

## 26.2 軍事

- 騎馬は平地で強い

- 足軽は防御と地形対応に優れる

- 鉄砲は距離を確保した場合に強い

- 単一兵科のみで常に勝てないようにする

- 数の差だけで勝敗を決定しない

## 26.3 外交

- 強大な勢力ほど周辺国から警戒される

- 約束違反には継続的な不利益を与える

- 遠方勢力による不合理な宣戦を抑制する

- 臣従は滅亡以外の選択肢として機能させる

# 27. 試作版の完了条件

以下をすべて満たした時点で、縦切り試作を完了とする。

- 3勢力からプレイ勢力を選択できる

- 10城がマップ上に表示される

- 時間停止と速度変更ができる

- 内政命令が数値へ反映される

- 武将が城間を移動する

- 軍団を編成して出兵できる

- AI勢力が内政、外交、軍事行動を行う

- 宣戦、同盟、臣従が機能する

- 攻城戦が機能する

- 3兵科による合戦が成立する

- 勝利条件と敗北条件が機能する

- セーブとロード後に状態が一致する

- 主要画面をスマートフォンで操作できる

- 致命的な型エラー、実行エラーが存在しない

- 1周30～60分で決着する

- 少なくとも3回の通しプレイが完了する

# 28. 評価項目

試作完成後、以下を評価する。

- 内政が単純作業になっていないか

- 外交が戦争回避や逆転に役立つか

- 武将の能力差が判断へ影響しているか

- 行軍を待つ時間に意味があるか

- 合戦で地形と兵科を使い分けられるか

- AIの行動を理解できるか

- 一つの最適戦法に固定されていないか

- 小勢力でも勝利可能か

- UI操作に迷う箇所がないか

- もう一度別勢力で遊びたいと思えるか

# 30. Codex運用原則

- 一度の指示で実装する機能は一つに限定する

- 実装前に関連ファイルを確認させる

- 実装後にテスト、型検査、ビルドを実行させる

- 完了条件を満たさない場合は完了扱いにしない

- 不明な仕様をCodexが独断で追加しない

- 仮データと正式データを明確に区別する

- UI変更後は画面表示を確認する

- バグ修正時は再発防止テストを追加する

- 仕様変更時はGDDと技術文書も更新する

- 大規模な全面改修より、小さな変更を積み重ねる

# 31. v0.1で固定する決定

- 一人用ゲームとする

- オフラインを基本とする

- 買い切り型を基本とする

- Web版から開発する

- Phaser、React、TypeScriptを使用する

- 縦切り試作は10城、30武将、4勢力とする

- リアルタイム進行と一時停止を採用する

- 外交、臣従、従属勢力を中核機能とする

- 合戦では地形と兵科相性を重視する

- ガチャ、スタミナ、PvPは実装しない

- 最初から全国版を作らない
